# Research report — agent platforms: persona · subagent · cron · mission

**Date:** 2026-06-29  
**Method:** heavy multi-agent deep-research — 6 parallel landscape researchers, 3-vote adversarial verification per load-bearing claim (≥2/3 refutes kills it), gap analysis, synthesis.  
**Scale:** 338 agents, 12.8M tokens. Findings total: 141; survived verification: 87; killed: 23.

> Feeds `2026-06-29-get-missions-done-epic.md`. Only claims that survived adversarial verification are treated as load-bearing.

## Gap analysis — the white space

### Gap 1: DURABLE MISSION STATE ACROSS CRON TICKS — a persona's scheduled job that resumes/advances its OWN mission file milestone-by-milestone, checkpointing progress to disk each tick. This is the sharpest white space: it is the literal intersection of all four primitives and NO platform does it. Today every scheduled run is a fresh, amnesiac, single-shot turn; every long mission is one in-context sequential burn that dies if the process dies.
- **Why unmet:** The combination defeats the dominant industry pattern. Factory Missions runs long but is NOT scheduled and NOT resumable across process restarts (it externalizes state to artifacts WITHIN one mission run, but a 16.5h run is one continuous orchestrated session, 4 validation rounds in-flight). Codex Automations are scheduled but each run lands in a review queue as a discrete unit — no carried mission state. ChatGPT/Gemini scheduled tasks are stateless check-ins. The Temporal/Kapoor-Narayanan findings name the unsolved primitive precisely: irreversibility + no rollback/recovery layer, and cascading step-failure (85% per-step => ~20% over 10 steps). A mission that advances ONE validated, disjoint milestone per cron tick — checkpointing [ ]->[x] to the persona-workspace mission file, with each milestone a fresh-context worker — converts a fragile 20-step in-context chain (compounds to ~20% success) into 20 independent 1-2 step runs that each succeed at ~95% and PERSIST. This is reliability-by-architecture, which the findings unanimously say is the binding constraint, not model capability. Done is uniquely positioned: it is the only product where the scheduler, the persona workspace (where missions live), and the milestone format already share one storage substrate.
- **Evidence:** Verified absent in live source: grep for 'mission' in harness/jobs/ returns ONLY permission-gate/lock comments — cron has ZERO mission awareness (the executor dispatches AgentTurn/Reminder payloads only; harness/jobs/model.py Payload union has no Mission payload). JobState (model.py:34-43) persists run-level health (last_status, consecutive_errors) but NO mission-progress/milestone-cursor field. The missions spec itself (line 26-33) lists 'Headless --mission -f auto-approve runner' and 'real worker/validator subagent spawning' as explicit Phase-2 non-goals, and the data-flow (line 108-121) is single-agent in-context sequential. The seam is designed (mission file = handoff surface, disjoint milestones, validate: field) but the cron<->mission<->subagent bridge is unbuilt. executor.py:182 already stamps the subagent wall-budget on scheduled turns, proving the runtime plumbing is 90% there — the missing piece is a Mission payload + a milestone-cursor in JobState + checkpoint-after-each-milestone.

### Gap 2: THE ASYNC SUPERVISION / HANDOFF SURFACE — when an unattended (cron) mission hits ambiguity, a risky/irreversible action, or low self-confidence, it must PAUSE the mission, persist a checkpoint, and escalate to a human review/approval queue, then RESUME from that checkpoint on approval. Done has no such surface; the only async-out channel is a notify() that just logs.
- **Why unmet:** Every finding converges on this as THE core unmet product need, and no platform ships it cleanly for the scheduled+autonomous case. Whitehead: 'the real skill isn't building the agent — it's designing the handoff' (fully-delegate vs quick-review vs human-only). Kapoor/Narayanan: calibration (knowing when you're wrong) is where even Opus 4.5/Gemini 3 score 25-52%; tail-failure, not average success, blocks unsupervised completion. The headless-cron findings name the exact failure: an interactive prompt ('confirm first') fed to a scheduled run STALLS because nobody answers (Done's stamp_headless_gate uses has_elicitation=False, so it deny-by-defaults file ops — safe, but a deny is a dead-end, not an escalation). The Fortune email-agent anecdote ('I had to RUN to my Mac Mini like defusing a bomb') is precisely the missing pause-checkpoint-escalate loop. Done's missions prose-gate (approve/edit/cancel) ALREADY exists for INTERACTIVE missions — the white space is making that same gate work ASYNCHRONOUSLY for a scheduled mission: serialize the decision into a durable review queue (analogous to Codex's review queue but mid-mission, not post-run), notify the human, and resume on their answer. Done's two-process boundary + content-gated tools + per-job Grant make it the right place to draw a per-milestone autonomy boundary (auto-advance read-only/working-tree milestones; HOLD-and-escalate any milestone whose validate: or grant touches irreversible/network/push actions).
- **Evidence:** Verified in live source: the ONLY async-out channel in harness/jobs/executor.py is _default_notify (line 60-61) which just calls logger.info — no review queue, no inbox, no approval-resume. grep for 'review.?queue|handoff|inbox|pending|approve' across harness/jobs/ + subagent.py returns nothing but docstring mentions of notify/approve. The missions spec explicitly DEFERS 'DecisionPrompt hard gate' to Phase-1.5 (spec line 30-31) and the gate it describes is INTERACTIVE prose ('agent stops, asks approve/edit/cancel', line 117) — there is no design for the gate firing inside an unattended cron run. CostGate.max_consecutive_failures (model.py:32) auto-DISABLES a job on repeated failure (a dead-stop), which is the opposite of a recoverable escalate-and-resume.

### Gap 3: RECONCILING SUBAGENT WORKER RESULTS INTO DURABLE MISSION STATE (the validator loop) — a worker completes one milestone, an inline/dedicated validator checks its 'validate:' assertion, and the OUTCOME is written back to the mission file as a durable, auditable record (pass/fail + one-line result + which worker), so the orchestrator can spawn a 'fix milestone' on failure. Today the subagent digest is returned to the parent's context window and discarded.
- **Why unmet:** This is the Factory-Droid validation-contract loop (validators surface issues, orchestrator spawns fix-features = 34.4% of work in their reference mission) — but Factory does it only inside one continuous long run, not persisted across scheduled ticks or restarts, and it requires Agent-Readiness-Level-4 + a scriptable QA harness most repos lack (their documented 'Feature retry limit reached' failure mode). Anthropic's multi-agent guidance says workers should write outputs to a filesystem/artifact store and pass lightweight references back to avoid the 'game of telephone' — Done's subagent does the OPPOSITE today (digest funneled straight into parent context, nothing persisted). The white space is uniquely Done's because Done already HAS the durable artifact store the worker should write to: the persona-workspace mission file with its per-milestone validate: field and [ ]->[x] checkbox seam. Wiring the subagent result -> validate the milestone -> checkpoint [x]+result-line into the mission file closes the orchestrator/worker/validator loop ON PERSISTED STATE, which neither Factory (no persistence across runs) nor Anthropic Research (ephemeral, research-only, 'not for code') does. It also directly answers the Anthropic fan-out failure modes (vague handoffs => duplicated/gapped work) because the mission file's disjoint-by-construction milestones ARE the explicit per-worker objective+boundary contract.
- **Evidence:** Verified in live source: harness/tools/subagent.py _format_digest (line 109-117) builds a text blob returned as the tool 'output' (line 151) straight into the parent's conversation — grep for 'write|store|persist|save|artifact' in subagent.py finds NOTHING (only the tool-description string). The worker is built with memory_root=None and skill_roots=None (lines 89-90) — it is deliberately stateless/ephemeral. There is no validator role and no fix-spawn logic (grep 'validat' across subagent.py + jobs/ finds only create.py's gate-validation, unrelated). The missions spec NAMES this exact seam as deferred: 'a future spawn tool hands a worker exactly one milestone and writes one structured result back' (spec line 105-106) and 'per-milestone validate: field; run inline today; a dedicated validator agent [later]' (line 44) — the format is built, the reconciliation loop is not.

### Done's structural advantages
- Done already has THREE of the four primitives interlocking at runtime, not just on paper. The cron executor (harness/jobs/executor.py) builds a full persona-faithful agent AND stamps env._remaining_secs from job.cost.timeout_s (line 182-183), so a SCHEDULED turn's subagent workers already inherit a wall-time budget. The subagent tool shipped in #173 (harness/tools/subagent.py) with depth-1 enforcement, per-worker step+wall caps, sibling isolation, and content-gating. So persona + cron + subagent already compose. No competitor combines all of: file-based mutable persona, in-process subagent fan-out, OS-service cron, AND a gated milestone plan in one local product.
- Done already ships the 'boring reliability layer' the market findings call the defensible value, and ships it MORE completely than the cron-feature leaders. CostGate (harness/jobs/model.py: timeout_s, min_cadence_s, max_consecutive_failures) + Grant (per-job tools/paths/write/exec/network) + JobState.consecutive_errors + O_EXCL single-instance daemon lock (prior PR #155) directly implement the locking + budget-cap + scoped-write posture that Hidekazu/Anthropic-vault findings name as what makes a timed agent safe. ChatGPT Tasks (1/hr cap, no tools, no files), Gemini (10 actions), and Anthropic Managed Agents are thin schedulers by comparison.
- Done's permission-gate + path-confinement (PR #170/#171, harness/jobs/executor.py stamp_headless_gate: deny-by-default, fail-CLOSED, file ops confined to workspace roots) is the EXACT mitigation for the OpenClaw security crisis (CVE-2026-25253, 341 malicious ClawHub skills, 258k exposed instances, CNCERT 'fail-open file access' warning). Done is built fail-closed on the surface that destroyed the leading open-source comp. This is a credibility moat for the 'always-on agent you can trust on your machine' positioning.
- Done's missions spec (.worktrees/missions-spec) is shaped as a fan-out seam from day one: disjoint milestones, per-milestone 'validate:' field, file-as-only-handoff-surface, status lifecycle in frontmatter. This mirrors Factory Droid's orchestrator/worker/validator EXACTLY but ships single-agent-first, so the validation-contract and worker seams drop in without format rework. Done copied the highest-leverage half (planning/spec mode) of the one platform with a published 16.5h multi-agent reference mission.
- Bounded, auditable, file-based memory (MEMORY.md, content-gated load-on-demand per PR #116) directly matches the Hermes/OpenClaw curated-eviction pattern that the findings say beats unbounded vector stores for auditability — and memory is the SINGLE named structural gap for recurring/scheduled agents (Anadkat: 'memory is either nonexistent or fragile'). Done's persona = mutable workspace with persistent memory is the right substrate for cross-run context that no scheduled-agent competitor has solved.

## Verified findings (survived adversarial verification)

### AI agent personas, persistent memory, multi-persona/multi-agent workspaces, and chief-of-staff personal agents (2025-2026). The dominant pattern across the leading open-source personal agents (OpenClaw, Hermes) is strikingly close to the quiubo-harness design: a plain-text, file-based persona ("SOUL"/"IDENTITY"/"USER") plus a bounded markdown memory file, plus a cron/heartbeat loop for proactivity, run as an OS-service daemon. This validates harness's architecture (SOUL.md/IDENTITY.md/USER.md, MEMORY.md, harness-cron with launchd/systemd, "Bob" chief-of-staff persona). The market's hard lessons are equally relevant: (1) file-based agents with host access are a severe security surface — OpenClaw went from viral to "security crisis" in three weeks (CVEs, hundreds of malicious skills, 258k exposed instances); (2) persona personality is widely judged "entertainment, not utility" — utility/memory compounds, quirks wear thin; (3) bounded memory (vs unbounded vector stores) trades recall for auditability and forces explicit eviction; (4) proactive/heartbeat agents risk notification fatigue; (5) multi-agent fan-out adds large token/latency overhead and can REDUCE accuracy vs a single agent. Net read for harness: the architecture is on-trend and defensible, but the differentiators worth leaning into are confinement/permission-gating (which harness PR #170 is already building), curated bounded memory, and restraint in both persona-flavor and proactive messaging.
- **[how-it-works, high]** OpenClaw (formerly Clawdbot/Moltbot), a leading open-source self-hosted personal agent, defines agent identity in a plain-markdown SOUL.md file that the agent reads on every wake (cron or user message) — 'it literally reads itself into existence' — directly paralleling harness's SOUL.md/IDENTITY.md persona files.  
  ↳ https://skywork.ai/skypage/en/openclaw-soul-ai-agent-identity-memory/2036774307250438144
- **[how-it-works, high]** OpenClaw stores long-term memory as plain Markdown: MEMORY.md holds durable learned facts about the user (e.g. 'user prefers concise responses', 'never schedule meetings on Fridays'), while per-day memory/YYYY-MM-DD.md logs are append-only and retrieved on demand via memory tools rather than injected every turn — mirroring harness's load-on-demand memory menu (PR #116).  
  ↳ https://bibek-poudel.medium.com/how-openclaw-works-understanding-ai-agents-through-a-real-architecture-5d59cc7a4764
- **[how-it-works, high]** OpenClaw runs a 'heartbeat' (cron-triggered agentic loop) that fires every 30 minutes by default; on each tick the agent reads a HEARTBEAT.md checklist, acts if needed, and otherwise replies HEARTBEAT_OK which the Gateway suppresses — the same proactive-cron pattern harness ships as harness-cron + first-run opt-in.  
  ↳ https://bibek-poudel.medium.com/how-openclaw-works-understanding-ai-agents-through-a-real-architecture-5d59cc7a4764
- **[how-it-works, high]** OpenClaw supports multi-agent routing: inbound channels/accounts/peers are routed to isolated agents, each with its own workspace and per-agent sessions — the same 'fleet of untyped mutable workspaces' model harness adopted for personas.  
  ↳ https://docs.openclaw.ai/
- **[how-it-works, high]** Nous Research's Hermes agent ('the agent that grows with you') uses two bounded memory files with hard char limits — MEMORY.md (2,200 chars / ~800 tokens, agent's own notes) and USER.md (1,375 chars / ~500 tokens, user profile) — stored in ~/.hermes/memories/ and injected into the system prompt as a frozen snapshot at session start.  
  ↳ https://hermes-agent.nousresearch.com/docs/user-guide/features/memory
- **[limitation, high]** Hermes memory deliberately does NOT auto-compact: when a write would exceed the char limit the memory tool returns an error rather than silently dropping entries, forcing the agent to consolidate/remove in the same turn before retrying — a curated-eviction design that trades recall breadth for auditability and bounded context cost.  
  ↳ https://hermes-agent.nousresearch.com/docs/user-guide/features/memory
- **[how-it-works, high]** Letta (formerly MemGPT) implements agent identity and user memory as two always-in-context 'core memory blocks' — a 'Persona' block (the agent's self-concept/personality) and a 'Human' block (facts about the user) — that the model self-edits in its loop; this is the same persona/user split as harness's SOUL.md (persona) + USER.md (user).  
  ↳ https://www.letta.com/blog/memory-blocks/
- **[how-it-works, high]** mem0 (a popular memory layer) does not store raw chat chunks: it runs an LLM extraction phase per message-pair to distill salient facts into compact natural-language memories, adding one LLM call per add() operation as overhead — the cost of fact-extraction memory vs simple file append.  
  ↳ https://mem0.ai/blog/state-of-ai-agent-memory-2026
- **[limitation, high]** OpenClaw's file-based transparency is explicitly framed as 'both its greatest engineering virtue and its biggest security surface' — an agent with access to your files, browser, email and messaging can be fully compromised by a malicious skill or a prompt injection from a web page it visits.  
  ↳ https://bibek-poudel.medium.com/how-openclaw-works-understanding-ai-agents-through-a-real-architecture-5d59cc7a4764
- **[limitation, med]** OpenClaw suffered a documented credential-theft CVE: CVE-2026-25253 (CVSS 8.8), where the Control UI accepted an unvalidated gatewayUrl query param and auto-opened a WebSocket transmitting the user's auth token; patched in v2026.1.29 (released 30 Jan 2026).  
  ↳ https://www.coral.inc/blog/2026-03-07-openclaw-security-crisis-2026
- **[limitation, med]** OpenClaw's skill marketplace (ClawHub) became a malware supply chain: Koi Security audited ~2,857 skills and flagged 341 malicious (335 from one coordinated 'ClawHavoc' operation); with no skill signing or mandatory review the marketplace was 'essentially an open door' — a warning for any agent that loads third-party skills.  
  ↳ https://www.coral.inc/blog/2026-03-07-openclaw-security-crisis-2026
- **[limitation, med]** By early March 2026 a passive internet scan found 258,305 publicly exposed OpenClaw instances (up from 42,665 in January) — the 'localhost fallacy': users deployed a locally-designed agent to cloud VPS for always-on availability, exposing port 18789 often with no auth (pre-v2026.1.29) or misconfigured trustedProxies.  
  ↳ https://www.coral.inc/blog/2026-03-07-openclaw-security-crisis-2026
- **[limitation, high]** China's CNCERT issued a national warning that OpenClaw's 'inherently weak default security configurations' plus its privileged system access could let attackers seize control of the endpoint, including via prompt injection embedded in web pages the agent reads — reinforcing that fail-open file/tool access is the central risk for personal agents (the exact gap harness PR #170 closes with permission-gating + path confinement).  
  ↳ https://thehackernews.com/2026/03/openclaw-ai-agent-flaws-could-enable.html
- **[how-it-works, high]** Claude Code custom subagents are file-defined personas (markdown in .claude/agents/ with a system prompt, scoped tool access, and independent context/permissions) invoked by name or auto-routed by description — confirming the file-based, per-persona-tool-scope model harness uses, from the platform harness is built on.  
  ↳ https://code.claude.com/docs/en/sub-agents

### Claude Code subagents (Task tool), Cursor's parallel-agent / plugin orchestration, and the LangGraph/CrewAI/AutoGen framework patterns — focused on how fan-out / handoff actually works, the verification and cost mechanics, and the live 2025-2026 debate over when multi-agent orchestration helps vs hurts. The strongest signal: the industry converged on a narrow winning pattern (single-threaded orchestrator + ephemeral, read-only/isolated subagents that return summaries) and away from peer-to-peer collaborating agents. Even the two loudest opposing voices — Anthropic ("build multi-agents") and Cognition ("don't build multi-agents") — actually agree on this orchestrator-worker shape; they disagree mostly on cost tolerance and parallelizability of the task domain.
- **[how-it-works, high]** Anthropic's Research feature uses an orchestrator-worker pattern: a lead agent (Claude Opus 4) plans, spins up 3-5 specialized subagents (Claude Sonnet 4) in parallel that each search with their own context window, then synthesizes findings with a separate citation pass.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[how-it-works, high]** Anthropic measured the multi-agent system (Opus 4 lead + Sonnet 4 subagents) outperforming single-agent Opus 4 by 90.2% on their internal research eval, and found token usage alone explains 80% of the performance variance on BrowseComp.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[limitation, high]** Anthropic states agents use ~4x more tokens than chat and multi-agent systems use ~15x more tokens than chat, and explicitly warns these architectures 'burn through tokens fast' and need tasks valuable enough to justify the cost.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[limitation, high]** Anthropic explicitly scopes WHERE multi-agent fails: most coding tasks have fewer truly parallelizable subtasks than research, and LLM agents 'are not yet great at coordinating and delegating to other agents in real time' — multi-agent excels only at heavy-parallelization, beyond-single-context, many-complex-tools tasks.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[limitation, high]** Anthropic's lead agents run subagents synchronously (waiting for each batch), which creates bottlenecks: the lead can't steer in-flight subagents, subagents can't coordinate with each other, and the whole system blocks on the slowest subagent; async execution is noted as future work that adds state-consistency and error-propagation problems.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[limitation, high]** Anthropic documents concrete fan-out failure modes from early versions: agents spawning 50 subagents for a simple query, subagents duplicating each other's work (e.g. two subagents both investigating 2025 chip supply chains) when the lead's task descriptions were too vague — so they hard-coded scaling rules (simple fact-find = 1 agent/3-10 calls; comparison = 2-4 subagents; complex = 10+).  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[how-it-works, high]** Anthropic's handoff-contract guidance: every subagent needs an explicit objective, output format, tool/source guidance, and clear task boundaries, or 'agents duplicate work, leave gaps, or fail' — vague instructions like 'research the semiconductor shortage' caused misinterpretation and duplicated searches.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[how-it-works, high]** Anthropic recommends subagents write outputs to a filesystem/artifact store and pass lightweight references back, rather than funneling everything through the lead, to avoid the 'game of telephone' — especially for structured outputs like code and reports.  
  ↳ https://www.anthropic.com/engineering/multi-agent-research-system
- **[limitation, high]** Cognition explicitly endorses Claude Code's pattern as correct precisely because its subagent does investigative work that stays OUT of the main agent's history (longer traces) and 'is usually only tasked with answering a question, not writing any code' — and warns running multiple parallel subagents would give 'conflicting responses.'  
  ↳ https://cognition.com/blog/dont-build-multi-agents
- **[how-it-works, high]** Cursor 2.0 (Oct 29, 2025) runs multiple agents in parallel without interference via git worktrees or remote machines, paired with the Composer model marketed as '4x faster than similarly intelligent models' to make parallel execution practical.  
  ↳ https://cursor.com/blog/2-0
- **[limitation, high]** Cursor's parallel-agent feature is Best-of-N, NOT orchestration: agents run the SAME prompt in isolated worktrees and race; the user picks one winner (Cursor 2.2 added Multi-Agent Judging to recommend a winner). Agents do not collaborate or hand off, which surprises developers expecting a Claude-Code-style manager/worker/synthesis flow.  
  ↳ https://www.elegantsoftwaresolutions.com/blog/cursor-multi-agent-not-what-you-think
- **[how-it-works, high]** Cursor 3 (April 2026) reframes the IDE around managing parallel agents, adds local-to-cloud agent handoff and multi-repo parallel execution, and ships a plugin marketplace letting teams extend agents with MCPs, skills, and subagents (plus private internal marketplaces).  
  ↳ https://www.infoq.com/news/2026/04/cursor-3-agent-first-interface/
- **[how-it-works, high]** LangGraph's supervisor pattern uses a central coordinator that routes each sub-task to a worker agent based on runtime state, while its swarm pattern lets agents hand off directly to each other via Command(goto=..., graph=Command.PARENT) objects from handoff tools — the supervisor trades latency for routing accuracy and easier debugging.  
  ↳ https://dev.to/focused_dot_io/multi-agent-orchestration-in-langgraph-supervisor-vs-swarm-tradeoffs-and-architecture-1b7e

### Devin, OpenAI Codex agent, and Claude Code as autonomous long-running coding agents — how they plan/execute multi-step missions, schedule/background runs, and where they fail (drift, supervision, trust) in 2025-2026
- **[how-it-works, high]** Devin's workflow is plan-then-execute with human gates: you describe the task, Devin builds a step-by-step plan you approve or redirect, then it executes autonomously in real time while you can intervene to redirect/correct at any point.  
  ↳ https://awesomeagents.ai/reviews/review-devin/
- **[how-it-works, high]** Cognition states Devin's sweet spot is 'tasks with clear, upfront requirements and verifiable outcomes that would take a junior engineer 4-8 hrs' (migrations, vuln fixes, unit tests, small tickets); it is 'infinitely parallelizable' across repos via a 'fleet of Devins'.  
  ↳ https://cognition.ai/blog/devin-annual-performance-review-2025
- **[limitation, high]** Cognition explicitly admits Devin 'can't independently tackle an ambiguous coding project end-to-end like a senior engineer could' — it needs specifics (component structure, color codes, spacing) and 'when outcomes aren't straightforwardly verifiable, additional human review is necessary.'  
  ↳ https://cognition.ai/blog/devin-annual-performance-review-2025
- **[how-it-works, high]** Cognition reports concrete throughput wins on bounded migration work: a bank migrated ETL files in 3-4 hours vs 30-40 for humans (10x), and an Oracle Java migration ran 14x faster than human engineers per repo.  
  ↳ https://cognition.ai/blog/devin-annual-performance-review-2025
- **[limitation, med]** Independent 2026 review: Devin 'does not understand trade-offs the way experienced engineers do... cannot evaluate whether a pattern is appropriate for your specific context. Architectural decisions still require human judgment.'  
  ↳ https://www.idlen.io/blog/devin-ai-engineer-review-limits-2026/
- **[how-it-works, high]** OpenAI describes Codex's coherence on long tasks as coming from a structured agent loop, not one giant prompt: Plan -> Edit code -> Run tools (tests/build/lint) -> Observe results -> Repair failures -> Update docs/status -> Repeat, which supplies real feedback, externalized state, and steerability.  
  ↳ https://developers.openai.com/blog/run-long-horizon-tasks-with-codex
- **[how-it-works, high]** Codex cloud runs each task in its own isolated microVM/sandbox preloaded with the repo; tasks typically complete in 1-30 minutes depending on complexity, and progress is monitored in real time.  
  ↳ https://developers.openai.com/codex/cloud
- **[limitation, high]** During Codex cloud task execution internet access is disabled — the agent is limited to the GitHub repo and dependencies pre-installed via a setup script, which prevents code exfiltration/supply-chain attacks but blocks tasks needing live external APIs/network.  
  ↳ https://developers.openai.com/codex/cloud
- **[how-it-works, high]** Codex relies on an AGENTS.md file (auto-loaded into context) to encode repo layout, build/test/lint commands, conventions, do-not rules, and 'what done means and how to verify work'; without accurate AGENTS.md guidance the agent repeats mistakes, and OpenAI advises adding rules only after observing repeated errors.  
  ↳ https://developers.openai.com/codex/learn/best-practices
- **[how-it-works, high]** Codex supports scheduled background work via Automations (instructions + optional skills running on a schedule), parallel threads across projects, and git worktrees to isolate runs; when an Automation finishes, results land in a review queue for a human to continue.  
  ↳ https://developers.openai.com/blog/run-long-horizon-tasks-with-codex
- **[how-it-works, high]** Anthropic frames long-running Claude as a shift from a tight conversational leash to specifying a high-level objective and setting 'a team of agents loose to work autonomously' — viable only where work is well-scoped, success criteria are clear, and human oversight can be 'occasional rather than continuous.'  
  ↳ https://www.anthropic.com/research/long-running-Claude
- **[how-it-works, high]** Claude Code's long-running pattern uses a CLAUDE.md file that Claude keeps in context as the living plan and can edit as it works; Anthropic's C-compiler project ran Claude across roughly 2,000 sessions to build a compiler that compiles the Linux kernel.  
  ↳ https://www.anthropic.com/research/long-running-Claude
- **[limitation, high]** Even in a successful autonomous Claude run, Anthropic's own author reports drift/limitations: clear gaps in test coverage (for a while testing only a single fiducial parameter point, 'drastically reducing its bug-catching surface area'), and 'elementary mistakes, such as tripping over gauge conventions or spending hours chasing bugs that a cosmologist would spot instantly.'  
  ↳ https://www.anthropic.com/research/long-running-Claude
- **[limitation, med]** Academic work characterizes two long-running failure modes: 'Safety Drift' (gradual erosion of declared safety intent leading to constraint-violating actions) and 'Operational Hallucination' (persistent repetitive tool calls indicating flawed state perception).  
  ↳ https://commons.clarku.edu/sops_fac/14/
- **[limitation, med]** Coding agents exhibit 'spiraling hallucination loops' where small deviations from reality compound — e.g., on a failed file read the agent fills the gap with a best guess and builds further reasoning on the false foundation rather than recognizing missing data.  
  ↳ https://surgehq.ai/blog/when-coding-agents-spiral-into-693-lines-of-hallucinations
- **[gap, med]** The dominant pragmatic pattern from developers extracting real value is orchestrating multiple bounded workflows with clear task boundaries, explicit handoffs, and deterministic checks around the model — not pursuing full end-to-end autonomy.  
  ↳ https://www.developersdigest.tech/blog/what-hacker-news-gets-right-about-ai-coding-agents-2026
- **[limitation, med]** A 2026 reliability analysis describes the core production gap as silent degradation: agents that work in demos but 'silently degrade as conversation complexity increased', hallucinate after ~step four of a workflow, or 'loop indefinitely when a caller deviates from the expected script' — the failure is architecture/oversight, not raw model quality.  
  ↳ https://www.inovabeing.com/blog/ai-agent-reliability-production-failure-2026

### Factory Droid Missions multi-agent orchestrator/worker/validator with milestones, validation contracts, and spec mode — exactly how it works, what users praise, what they complain about, and explicit limitations (2025-2026)
- **[how-it-works, high]** Missions uses a three-role architecture: an orchestrator that plans/decomposes work and steers it (never building directly), workers that implement well-specified features, and validators that evaluate completed work for correctness — validators surface bugs/gaps but do NOT fix them; they hand issues back to the orchestrator, which spawns 'fix features' for future workers.  
  ↳ https://factory.ai/news/missions-architecture
- **[how-it-works, high]** The orchestrator defines correctness FIRST by writing a 'validation contract' (a set of behavioral assertions defining success) before defining any features, so the contract is derived from requirements rather than from already-planned implementation. These assertions are later checked by fresh agents that exercise the system as a black box, like a real user, not by inspecting the code.  
  ↳ https://factory.ai/news/missions-architecture
- **[how-it-works, high]** Missions applies test-driven development at two levels: each worker writes tests before code (worker level), and the orchestrator writes the validation contract before features (mission level).  
  ↳ https://factory.ai/news/missions-architecture
- **[how-it-works, high]** State is externalized/distributed across shared artifacts (validation contract, feature list, research notes, operational guidelines, evolving knowledge base) so no single agent holds the whole project in its context window; each agent reads only what is relevant to its current job.  
  ↳ https://factory.ai/news/missions-architecture
- **[how-it-works, high]** An orchestrator breaks a project into milestones (each a meaningful checkpoint); every milestone ends with a validation phase (run tests, check regressions, verify integration), and each feature within a milestone gets a fresh worker session with clean context. Missions parallelizes within features and during validation where coordination overhead is low.  
  ↳ https://factory.ai/news/missions
- **[how-it-works, high]** Spec Mode is invoked with the --use-spec flag (e.g. 'droid exec --use-spec --auto low "refactor the auth module"'); the agent plans before executing, and you can use a separate, cheaper model for the spec phase via --spec-model (e.g. --spec-model claude-haiku-4-5-20251001).  
  ↳ https://docs.factory.ai/cli/features/missions
- **[how-it-works, high]** A user starts a mission by running '/missions' in any Droid session; Droid then has a back-and-forth conversation (clarifying questions, probing constraints) to build a structured plan of features organized into milestones, before handing off to the orchestration layer (Mission Control).  
  ↳ https://docs.factory.ai/cli/features/missions
- **[how-it-works, high]** Model specialization is built in: once roles are separated, model choice is local to each role — broad planning/judgment models for the orchestrator, reliable/cost-efficient models for workers, and thorough/skeptical models for validators.  
  ↳ https://factory.ai/news/missions-architecture
- **[how-it-works, high]** In Factory's published Slack-clone reference mission (16.5h runtime), validators surfaced 81 total issues (65 blocking, 11 non-blocking, 5 suggestions) and the orchestrator generated 21 fix features against 40 original features — fix features were 34.4% of implementation work (21/61). Time split: orchestration 2.3%, implementation 60.5%, validation 37.2%.  
  ↳ https://factory.ai/news/missions-architecture
- **[how-it-works, high]** That same reference mission consumed 778.5M total tokens (744.9M of which were cache reads), produced 38.8k lines of code (52.5% tests) at 89.25% statement coverage, across 185 agent runs, and required 4 validation rounds to pass all 6 milestones (Round 1: 0/6, Round 2: 1/6, Round 3: 2/6, Round 4: 6/6).  
  ↳ https://factory.ai/news/missions-architecture
- **[limitation, med]** The orchestrator prompt characterizes workers as a known weak link: 'Implementation workers are skilled and efficient and execute well-specified features well, but struggle with ambiguity and can be lazy' — so the orchestrator is told to be explicit about context, constraints, and acceptance criteria.  
  ↳ https://gist.github.com/V1ki/356b121038722ebf32b5aac85482c113
- **[limitation, high]** Missions is explicitly shipped as a research preview with open, unresolved questions Factory itself poses: whether parallelization actually beats sequential execution, how to maximize correctness as long-running plans accumulate errors, and the cost-vs-quality tradeoff ('more planning and validation means higher cost but potentially better output').  
  ↳ https://docs.factory.ai/cli/features/missions
- **[limitation, high]** Missions requires High autonomy or --skip-permissions-unsafe (which skips all permission checks and is intended only for isolated sandboxes); admins can restrict who is allowed to start Missions.  
  ↳ https://docs.factory.ai/cli/features/missions
- **[limitation, high]** For Missions' self-validating QA to work, the repository should be at Agent Readiness Level 4 (Optimized) or above, and the codebase must provide an automated, scriptable way to exercise the app like a real user (e.g. a script to stand up or mock all dependencies). Without it, the mission cannot reliably verify its own work.  
  ↳ https://docs.factory.ai/cli/features/missions
- **[limitation, high]** A documented failure mode is 'Feature retry limit reached': a single feature fails repeatedly and the orchestrator stops retrying — Factory says this is often a user-testing validator failing because the app cannot be QA'd correctly (no reliable way to stand up/exercise the app), fixable by providing a dependency-mocking script or by disabling QA validation in Mission Control settings.  
  ↳ https://docs.factory.ai/cli/features/missions
- **[limitation, high]** Third-party review (digitalapplied) frames Factory's existence of a separate review droid as evidence the platform does NOT expect autonomous ship-to-production: the code droid's output needs scrutiny, human merge authority should stay, and long-running multi-step autonomy 'remains an open industry problem' that Factory handles with checkpoints rather than pretending it is solved.  
  ↳ https://www.digitalapplied.com/blog/factory-ai-multi-agent-coding-platform-review

### Scheduled / cron / always-on autonomous AI agents (2025-2026): the capability shipped across every major assistant as a thin scheduling layer over the same fragile agent loop. The honest story is a sharp split between marketing ("works while you sleep") and operator reality ("a toddler that needs to be overseen"). The hard, verifiable wins are in the boring infra — schedulers, credential vaults, locks, budget caps — not in autonomy itself, which still degrades over long horizons and demands human checkpoints. For a product like this harness (which ships its own cron daemon), the differentiated, defensible value is exactly that boring reliability layer (single-instance locking, budget/turn caps, scoped write permissions, review loops), not the promise of unattended autonomy.
- **[how-it-works, high]** Every major assistant now ships a recurring-schedule feature as a thin layer over its existing agent: OpenAI 'Scheduled Tasks' in ChatGPT, Google Gemini 'Scheduled Actions', and Anthropic 'Managed Agents' scheduled deployments — confirming this is a table-stakes capability category, not a differentiator.  
  ↳ https://help.openai.com/en/articles/10291617-tasks-in-chatgpt
- **[limitation, high]** ChatGPT Scheduled Tasks are capped at running at most once per hour, and active-task counts are tightly limited by plan tier (Go 3, Plus 5, Business/Edu 10, Pro/Enterprise 15) — these are recurring check-in tools, not general-purpose schedulers.  
  ↳ https://help.openai.com/en/articles/10291617-tasks-in-chatgpt
- **[limitation, med]** ChatGPT Scheduled Tasks explicitly cannot run custom GPTs/tools, cannot use webhooks (so they suit time-based recurrence but NOT event-triggered automation), and cannot access uploaded project files — sharp functional gaps vs a real automation engine.  
  ↳ https://www.usecarly.com/blog/chatgpt-scheduled-tasks/
- **[how-it-works, high]** Google Gemini Scheduled Actions let users phrase a recurring prompt in natural language; it runs in the background and notifies on completion, but is capped at 10 active actions and gated behind Google AI Pro/Ultra or qualifying Workspace plans.  
  ↳ https://support.google.com/gemini/answer/16316416
- **[how-it-works, high]** Anthropic shipped Managed Agents 'scheduled deployments' (cron) plus a credential vault in a public beta on ~June 9, 2026 — explicitly reframing an AI agent as 'a cron job with a credential vault' that 'does not need you awake.'  
  ↳ https://www.techtimes.com/articles/318163/20260610/claude-managed-agents-add-cron-schedules-credential-vaultsanthropic-beta-puts-agents-autopilot.htm
- **[how-it-works, high]** Anthropic's vault design keeps API keys away from the model: the sandbox holds only a placeholder env-var name plus an allow-listed domain list, and the real secret is injected at the network boundary only for allowed domains — a concrete mitigation against prompt-injection credential exfiltration in unattended runs.  
  ↳ https://www.techtimes.com/articles/318163/20260610/claude-managed-agents-add-cron-schedules-credential-vaultsanthropic-beta-puts-agents-autopilot.htm
- **[how-it-works, high]** The failure mode unique to scheduling is overlap: a slow run still going when the next tick fires means two instances race on the same repository. The documented fix is a mutual-exclusion lock (`flock -n` on one host, or a CI concurrency group) PLUS idempotency/stable session IDs — locking + idempotency is what makes a timed agent safe to leave on indefinitely.  
  ↳ https://hidekazu-konishi.com/entry/claude_code_cicd_and_headless_automation.html
- **[limitation, high]** Unattended cost control rests on hard caps because there is no human to stop a loop: `--max-turns` bounds iterations, `--max-budget-usd` bounds spend, plus CI concurrency limits and wall-clock timeouts — a runaway/looping job should fail rather than spend.  
  ↳ https://hidekazu-konishi.com/entry/claude_code_cicd_and_headless_automation.html
- **[limitation, high]** For unattended runs the recommended permission posture is restrictive: use `dontAsk` (any non-allow-listed tool call is auto-denied so nothing hangs on an approval no one will answer), and reserve `bypassPermissions`/`--dangerously-skip-permissions` strictly for disposable sandboxes — they even refuse to run as root, signaling how much trust they assume.  
  ↳ https://hidekazu-konishi.com/entry/claude_code_cicd_and_headless_automation.html
- **[limitation, high]** A documented recurring pitfall: feeding an interactive prompt ('let me know if...', 'confirm first') to a headless scheduled run stalls or aborts it because nobody is there to answer — headless/scheduled prompts must be complete, self-contained instructions with the success criterion stated up front.  
  ↳ https://hidekazu-konishi.com/entry/claude_code_cicd_and_headless_automation.html
- **[limitation, high]** Concrete failure anecdote: a user's email agent lost its original instruction in a real inbox (after weeks of working in a test inbox) and began acting on the entire inbox while ignoring instructions to pause and confirm — 'I had to RUN to my Mac Mini like I was defusing a bomb.' Test-environment success did not transfer to production.  
  ↳ https://fortune.com/2026/02/23/always-on-ai-agents-openclaw-claude-promise-work-while-sleeping-reality-problems-oversight-guardrails/
- **[limitation, high]** The core technical ceiling for long-running autonomy, per ex-OpenAI engineer Shyamal Anadkat: 'A system that's 95% accurate on individual steps becomes chaotic over a 20-step autonomous workflow' — long-horizon planning is still weak, so agents do well on short chains but fall apart on complex multiday projects.  
  ↳ https://fortune.com/2026/02/23/always-on-ai-agents-openclaw-claude-promise-work-while-sleeping-reality-problems-oversight-guardrails/
- **[gap, high]** Memory is a named gap for scheduled/recurring agents: 'In many agents, memory is either nonexistent or fragile. You need systems that can maintain a coherent model of your work context, priorities, and constraints' (Anadkat) — directly relevant to agents that must persist context across recurring runs.  
  ↳ https://fortune.com/2026/02/23/always-on-ai-agents-openclaw-claude-promise-work-while-sleeping-reality-problems-oversight-guardrails/
- **[gap, high]** The write/commit/push authority boundary must be drawn per scheduled job and enforced at the narrowest layer: read-only for triage/report, write-to-working-tree (no push/network) for fixes, and full-auto only in disposable sandboxes — the more irreversible the action (push/deploy/delete/external API call), the more it belongs outside the agent's tool surface with the pipeline mediating it.  
  ↳ https://hidekazu-konishi.com/entry/claude_code_cicd_and_headless_automation.html

### What is missing in AI agent products: autonomous mission completion, trust/supervision of long-running agents, combining scheduling + planning + delegation, why agents fail to "get things done" — analyst and founder takes, 2025-2026. Core synthesis: the binding constraint on 2025-2026 agents is NOT model capability but RELIABILITY-OVER-TIME and TRUST/SUPERVISION DESIGN. Models can do individual hard steps but "can't hold it together" across long-horizon multi-step missions (per-step error compounds; context/memory drifts; integrations break). The unsolved product layer is the human-agent HANDOFF — bounded autonomy, recoverable/reversible actions, and supervision UX — not better planning or delegation primitives, which already exist (planner-decomposes-delegates-to-workers is the standard pattern). The market gap: products that make long-running autonomy TRUSTWORTHY (auditable, interruptible, calibrated to its own uncertainty, with clean delegation boundaries), rather than more capable.
- **[limitation, high]** Gartner predicts over 40% of agentic AI projects will be canceled by the end of 2027, driven by escalating costs, unclear business value, or inadequate risk controls — not by a lack of raw model capability.  
  ↳ https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027
- **[gap, high]** Gartner attributes much of the agent market to 'agent washing' — rebranding chatbots, RPA, and assistants as agents — estimating only ~130 of the thousands of agentic-AI vendors are real, and says current models lack the maturity/agency to autonomously achieve complex business goals or follow nuanced instructions over time.  
  ↳ https://www.gartner.com/en/newsroom/press-releases/2025-06-25-gartner-predicts-over-40-percent-of-agentic-ai-projects-will-be-canceled-by-end-of-2027
- **[how-it-works, high]** Princeton researchers Sayash Kapoor and Arvind Narayanan (authors of 'AI Snake Oil') argue most agent benchmarks report only AVERAGE accuracy, which masks unreliability; their paper proposes evaluating reliability on four axes — consistency, robustness, calibration, and safety — broken into 14 metrics. This reframes the missing capability as reliability science, not capability.  
  ↳ https://fortune.com/2026/03/24/ai-agents-are-getting-more-capable-but-reliability-is-lagging-narayanan-kapoor/
- **[limitation, high]** On Kapoor/Narayanan's reliability benchmark, even the best models (Claude Opus 4.5 and Gemini 3 Pro, ~85% overall) had alarming sub-scores: Gemini 3 Pro scored only 52% at judging when its own answers were accurate (calibration) and 25% at avoiding catastrophic mistakes (safety); Claude Opus 4.5 was only 73% consistent. This shows the gap is uncertainty-awareness and bounded failure, not task skill.  
  ↳ https://fortune.com/2026/03/24/ai-agents-are-getting-more-capable-but-reliability-is-lagging-narayanan-kapoor/
- **[limitation, high]** The authors state reliability is a hard prerequisite specifically for AUTONOMY: 'an agent that succeeds on 90% of tasks but fails unpredictably on the remaining 10% may be a useful assistant yet an unacceptable autonomous system.' Unpredictable tail-failure — not average success — is what blocks unsupervised mission completion.  
  ↳ https://fortune.com/2026/03/24/ai-agents-are-getting-more-capable-but-reliability-is-lagging-narayanan-kapoor/
- **[how-it-works, high]** Temporal frames the core failure as cascading step-failure: even at 85% per-step reliability, a 10-step workflow succeeds end-to-end only ~20% of the time, and longer production workflows degrade further — strong per-step performance still produces compounding end-to-end failure. This is the math behind why agents fail at long autonomous missions.  
  ↳ https://temporal.io/blog/ai-reliability-is-a-decade-old-problem
- **[gap, high]** Temporal argues the new dimension is irreversibility: when an agent hallucinates mid-workflow it can wipe a drive, make an unauthorized purchase, or fabricate records, and many agent stacks have 'no mechanism to recover' from such irreversible actions — a missing rollback/recovery layer, not a missing model capability.  
  ↳ https://temporal.io/blog/ai-reliability-is-a-decade-old-problem
- **[gap, high]** AI-ops consultant Breeanna Whitehead names the missing product primitive directly: 'The real skill isn't building the agent — it's designing the handoff.' Most users either over-trust agents and clean up messes or micromanage them; the gap is clear handoff points (fully delegate vs. quick-review vs. human-only). This is the supervision/delegation-boundary gap framed as the core unmet need.  
  ↳ https://fortune.com/2026/02/23/always-on-ai-agents-openclaw-claude-promise-work-while-sleeping-reality-problems-oversight-guardrails/
- **[limitation, high]** Where agents already deliver value is a narrow 'middle layer' of knowledge work that used to eat 2-3 hours/day — synthesizing meeting notes into action items, drafting follow-ups in someone's voice, research briefs, organizing priorities into a plan — while tasks needing judgment, ambiguity, or relationships still fail. This bounds where autonomous completion actually works today.  
  ↳ https://fortune.com/2026/02/23/always-on-ai-agents-openclaw-claude-promise-work-while-sleeping-reality-problems-oversight-guardrails/
- **[how-it-works, high]** An academic benchmark of 34 programmable tasks across three open-source agent frameworks plus two LLM backbones measured only ~50% task completion, and derived a failure taxonomy of 19 distinct causes across three tiers aligned to task phases: task planning (improper decomposition, failed self-refinement / infinite retry loops, unrealistic planning), task execution (tool-use failures, faulty code generation), and incorrect response generation.  
  ↳ https://arxiv.org/html/2508.13143v1
- **[limitation, high]** Planning failures specifically include 'failed self-refinement' — the agent cannot learn from its own past errors and repeats the same failed sub-task in an infinite loop — and 'unrealistic planning' — producing plausible steps that exceed downstream agents' actual capabilities. These are autonomy-killing loop/competence-mismatch failures distinct from raw model quality.  
  ↳ https://arxiv.org/html/2508.13143v1
- **[gap, med]** Memory/state is a named structural gap: bigger models and larger context windows do not fix forgetfulness — agents still can't reliably recall 'what a user told it last Tuesday' or that a prior recommendation turned out wrong — which is why 2026 sees purpose-built agent-memory layers (Mem0, Zep, Letta) emerge to handle cross-session persistence that raw vector stores never solved.  
  ↳ https://mem0.ai/blog/state-of-ai-agent-memory-2026
- **[gap, med]** McKinsey's AI Trust survey (~500 orgs, Dec 2025-Jan 2026, as reported) found only ~1 in 3 report mature strategy/governance/agentic controls, and security/risk is the top barrier to scaling agentic AI (cited by ~two-thirds) — i.e. the constraint on deployment is confidence in safe autonomous operation, not capability. The governance/oversight infrastructure lags the growth in agent surface area.  
  ↳ https://semanticos.io/blog/ai-trust-2026-agentic-era/
